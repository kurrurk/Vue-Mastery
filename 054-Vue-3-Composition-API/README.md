# Why the Composition API

🧩 В данном разделе рассмотрены причины появления Composition API и проблемы, которые он решает по сравнению с классическим Options API.

## Почему появился Composition API?

По мере роста приложения компоненты становятся всё больше и сложнее. Логика, относящаяся к одной функциональности, оказывается разбросана между различными секциями компонента (`data`, `computed`, `methods`, `watch`), что усложняет чтение и сопровождение кода.

Composition API позволяет группировать связанную логику в одном месте независимо от её типа.

## Проблемы Options API

- Логика одной функции разбросана по нескольким секциям компонента.
- Крупные компоненты становятся сложными для чтения.
- Сложнее сопровождать и изменять код.
- Повторное использование логики между компонентами затруднено.

### Пример

В компоненте поиска можно встретить следующую структуру:

```text
data
 ├── search
 └── results

computed
 └── filteredResults

watch
 └── search

methods
 ├── search()
 └── clearSearch()
```

Несмотря на то что всё относится к поиску, логика распределена по разным разделам компонента.

## Подходы к повторному использованию логики

До появления Composition API существовало несколько способов переиспользования кода.

### Mixins

Позволяют подключать общую логику сразу в несколько компонентов.

**Преимущества**

- Простое повторное использование кода.

**Недостатки**

- Конфликты имён.
- Неявное происхождение свойств и методов.
- Сложность сопровождения больших проектов.

### Higher-Order Components (HOC)

Компонент-обёртка добавляет дополнительное поведение существующему компоненту.

**Преимущества**

- Повторное использование логики.

**Недостатки**

- Увеличение количества компонентов.
- Более сложная структура приложения.

### Renderless Components

Компоненты содержат только логику и передают данные через слот, оставляя отображение родительскому компоненту.

**Преимущества**

- Разделение логики и представления.
- Высокая гибкость.

**Недостатки**

- Более сложная структура шаблонов.
- Увеличение вложенности компонентов.

## Решение — Composition API

Composition API позволяет вынести связанную функциональность в отдельные композиционные функции (Composables) и затем использовать её в любых компонентах.

### Преимущества Composition API

- Логика группируется по функциональности, а не по типу.
- Более компактные и понятные компоненты.
- Простое повторное использование кода.
- Улучшенная читаемость.
- Более удобное масштабирование крупных приложений.
- Упрощение тестирования отдельных частей логики.

## Практика

- Рассмотрены недостатки Options API в больших компонентах.
- Изучены существовавшие способы повторного использования логики (Mixins, Higher-Order Components и Renderless Components).
- Показано, каким образом Composition API объединяет преимущества предыдущих подходов и делает код более читаемым, гибким и удобным для сопровождения.
